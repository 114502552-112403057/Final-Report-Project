## 1. 專案評估指標設定（A.C.T.I.V.E）



| 指標代號 | 指標名稱 | 評估內容 | 量化方式 |
| -------- | -------- | -------- | -------- |
| A | Activity（活躍度）| Commit、PR、Issue 更新頻率 | 過去 30 天 Commit 數、PR 數、Issue 數 |
| C | Communication（溝通程度）| 維護者的回覆速度 | PR/Issue 平均首回應時間（小時） |
| T | Transparency（治理透明度）| 專案規範完整度 | 是否有 CONDUCT / CONTRIBUTING / RELEASE（0–3分） |
| I | Issue | 專案規範完整度 | 標籤分類清晰度 | Issue 標籤數量、分類覆蓋率 (%) RELEASE（0–3分） |
| V | Version Stability（版本穩定性） | Release 規律度、間隔是否穩定 | 過去 6 個月 Release 次數與變異數 |
| E | Ease of Contribution（貢獻易用性） | 上手難度、文件完整度 | 從 README →能跑起環境所需的步數與錯誤率數 |

## 2. 專案評估結果（[yt-dlp](https://github.com/yt-dlp/yt-dlp)）

yt-dlp 是影音下載 CLI 工具，支援多個影音平台、字幕 統合處理，並有高度模組化的架構。

| 指標 | 數據（量化觀察） | 分數 | 分析 |

| A – 活躍度 | Activity（活躍度）| 5/5 | 超高活躍度 |

### yt-dlp-wiki FAQ錯字、文法修正 (賴泉宏)

#### 1. 查看文件並審視規範
a.首先詳細閱讀 FAQ.md 的原始文件內容以尋找潛在問題。

b.接著仔細研讀並理解專案的貢獻規範 CONTRIBUTING.md。

c.確認並遵守 AI/LLM 使用政策，確保內容為人工校對而非直接生成。

* 審閱過程中發現的問題：
    1. 在 File name too long 章節發現重複單字 the the。
    2. 在指令範例中發現明顯拼字錯誤 comamnd
    3. 在多處發現語法上的代名詞誤用（whom vs which）與不當的大寫（Many）。

#### 2. 準備階段與環境建置
* 確認錯誤性質為單純的文件修正（Typos），決定直接進行修復：
    a.將官方的文件儲存庫 yt-dlp/yt-dlp-wiki Fork 到個人的 GitHub 帳號下。
    
    b.在個人的儲存庫中建立一個新的分支（Branch），以隔離本次的修改內容，避免直接汙染主分支。
    
#### 3. 執行修正 (Fix Typos)
* 修正內容 : 
    a.刪除贅字，limit to "the the => the" length。
    
    b.修正拼字錯誤，rest of your comamnd => command。
    
    c.修正代名詞誤用，many of "whom => which"。
    
    d.修正大小寫，"many => Many of the issue reports。

#### 4. 發出 Pull Request (PR)
* PR : [FAQ] [Fix typos #66](https://github.com/yt-dlp/yt-dlp-wiki/pull/66)
* 將變更提交並建立 Pull Request：
    * 設定標題：docs : Fix typos and grammar in FAQ.md。
    * 在描述欄位中，清楚列出了三點具體的修改細節（1. limit to... 2. comamnd... 3. Many...），方便維護者快速對照。
    * ![image](https://hackmd.io/_uploads/Sy3GxYwZ-g.png)


#### 5. 與維護者互動與審核
* 提交 PR 後進入審核階段，與維護者 bashonly 和 pzhlkj6612 的互動過程：
    * 維護者 bashonly 主動協助將標題修改為更符合專案內部規範的格式：[FAQ] Fix typos。
    * 維護者很快就對變更進行了審核（Code Review）並給予批准（Approve）並且 merge。
